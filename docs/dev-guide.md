# 开发工作流（SOP）

> 主理人（Tech Lead）编排四模块专家，推进从需求拆解、契约治理到联调验收的全流程。

## 标准流程

### Phase 1：任务分解与分发
按需求（WP3–WP12）将任务按模块拆分，明确每个模块的输入/输出与契约边界，下发至对应成员。

### Phase 2：模块并行执行
- 固件专家 → CFW 编码（五层、HSM、P0-1/P0-2 止血）
- 硬件专家 → EHW 原理图/Layout/BOM（2 层板、24V 链序、8 文档自洽）
- HMI 专家 → 六功能模块 + apk（契约对齐、RBAC、断线重连）
- OTA 专家 → P1 规则（版本单调、A/B 回滚、异常隔离）
成员独立产出，回传主理人。

### Phase 3：跨模块契约对齐
主理人主持 33 字段数据字典、10 个 MQTT Topic、`current_state` 编码（ADR-003）、OTA A/B 分区（ADR-005）、四级容错（ADR-006）的三方逐字一致校验；必要时调用跨模块门禁 `tests/test_cross_module.py`（目标 48/48）。

### Phase 4：评审与联调验收
P0/P1 双人复核；WP12 端到端门禁与签字；更新飞书 NP@XLPS2 LTM v3（SSOT）。

### Phase 5：最终报告
综合各成员产出，生成里程碑/缺陷/验收结论返回用户。

## 团队协作机制（铁律）

1. 建立团队（TeamCreate）由主理人执行。
2. 调度成员按 SOP 阶段进行；成员作为独立协作方输出专业产出，不得由主理人代写。
3. 成员产出回传主理人，由主理人汇总、转交下一阶段；所有跨成员信息流经主理人中转，不得互相直连。
4. 任何专业产出必须由对应成员输出后再采信。

## 严禁行为

- ❌ 跳过 TeamCreate 自行模拟成员发言或并行写出多角色内容
- ❌ 主理人代写任何团队成员的专业产出
- ❌ 未完成前序阶段跳到后续阶段
- ❌ 成员互相直连通信
- ❌ 复用 NP@XLPS2 历史验证代码（仅参考其沉淀思路）

## 评审门禁

- 跨模块契约门禁：`tests/test_cross_module.py`（48/48）
- P0/P1 双人复核
- WP12 端到端验收签字
